# Система запису на співбесіди

Веб-сайт для запису кандидатів на співбесіди + адмін-панель для рекрутерів.

- **Основний сайт** (`/`) — кандидат обирає свою ОП (освітню програму), бачить доступні часові слоти
  (підібрані автоматично системою) та записується, вказавши ПІБ, email, тег у Telegram та навчальну групу.
- **Адмін-панель** (`/admin`) — рекрутери вказують свій вільний час, система автоматично знаходить спільний
  вільний час пари рекрутерів (основний + додатковий) для кожної ОП і виставляє ці слоти на основний сайт.
  Є загальний календар співбесід та окремий календар для кожного рекрутера. Опціонально — інтеграція з
  Google Calendar (створення подій, врахування зайнятості).

## Технології

- Backend: Node.js (Express) + SQLite (вбудований `node:sqlite`, без нативної компіляції)
- Frontend: React (Vite) + react-big-calendar
- Авторизація: JWT + bcrypt
- Google Calendar API (опціонально)

## Структура проєкту

```
interview-scheduler/
├── backend/     — API сервер
└── frontend/    — веб-інтерфейс (React)
```

## Швидкий старт (локально)

### Вимоги

- Node.js **>= 22.5.0** (потрібен для вбудованого `node:sqlite`)

### 1. Backend

```bash
cd backend
cp .env.example .env
```

Відкрийте `.env` і заповніть значення:

- `JWT_SECRET` — **обов'язково** довгий випадковий рядок (наприклад, `openssl rand -hex 32`). Сервер не
  запуститься, якщо ця змінна не змінена зі значення-заглушки.
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` — облікові дані адміністратора, які будуть створені при першому запуску.
- `DEFAULT_RECRUITER_PASSWORD` — пароль, який буде встановлено для всіх початково створених акаунтів
  рекрутерів. Кожен рекрутер має змінити його після першого входу (Профіль → Змінити пароль).
- `FRONTEND_URL` — адреса фронтенду (для CORS та редиректів Google OAuth), за замовчуванням
  `http://localhost:5173`.
- Налаштування Google Calendar — див. розділ нижче (можна залишити порожніми, тоді інтеграція просто
  буде вимкнена).

Встановити залежності та запустити:

```bash
npm install
npm run dev
```

При першому запуску база даних `data.sqlite` створюється автоматично, а також виконується seed:
створюються 7 ОП (ІА, ІМ, ІП, ІС, ІО, ІК, ІІ), акаунт адміністратора та стартовий набір рекрутерів з
командами, як було вказано у вимогах. Усе це повністю редагується далі через адмін-панель
(розділи «Рекрутери» та «Команди по ОП»).

API запуститься на `http://localhost:4000`.

### 2. Frontend

У новому терміналі:

```bash
cd frontend
npm install
npm run dev
```

Відкриється на `http://localhost:5173`. Сторінка `/` — публічна сторінка запису, `/admin/login` — вхід в
адмінку.

## Облікові дані за замовчуванням

| Роль | Email | Пароль |
| --- | --- | --- |
| Адміністратор | значення `ADMIN_EMAIL` | значення `ADMIN_PASSWORD` |
| Рекрутери (Дарина, Ілюша, Артем, ...) | `<імʼя>@example.com` (див. `backend/src/db/seed.js`) | значення `DEFAULT_RECRUITER_PASSWORD` |

**Обов'язково змініть усі паролі після першого входу** (Профіль → Змінити пароль), або через
«Рекрутери» → «Пароль» в адмінці (для адміністратора).

## Як це працює

1. **Адмінка → «Мій вільний час»**: кожен рекрутер обирає дату та позначає галочками слоти у фіксованій
   сітці з 08:00 до 21:00 (кожен слот — 45 хв співбесіди + 15 хв перерви: 08:00–08:45, 09:00–09:45, ...,
   20:00–20:45). Можна обрати кілька слотів одразу і натиснути «Підтвердити вибір» — система збереже весь
   набір слотів на цей день за один раз (попередній вибір на цей день буде замінено новим).
2. **Адмінка → «Команди по ОП»**: для кожної ОП налаштовується список «основних» рекрутерів — тих, хто
   представляє цю ОП на співбесіді. Список «додаткових» позначає, що рекрутер входить до загального пулу
   потенційних партнерів. Це повністю редаговано — можна в будь-який момент змінити склад команд.
3. Система автоматично шукає спільний вільний час для кожної пари «основний рекрутер цієї ОП +
   будь-який інший активний рекрутер, що входить хоча б у одну команду» (необов'язково саме «додатковий»
   для цієї ж ОП — головне, щоб одна сторона пари представляла ОП кандидата). Спільний вільний час ділиться
   на слоти фіксованої тривалості (за замовчуванням 45 хв, налаштовується через
   `DEFAULT_SLOT_DURATION_MINUTES`) і виставляється як доступні слоти для відповідної ОП.
4. Кандидат на основній сторінці обирає ОП, бачить доступні слоти, заповнює форму (ПІБ, email, Telegram,
   група) і записується. Слот стає зайнятим і зникає зі списку доступних.
5. Записана співбесіда з'являється у «Загальному календарі» та в особистих календарях обох рекрутерів.
6. Якщо рекрутер підключив Google Calendar (Профіль → Google Calendar), для нього автоматично створюється
   подія у його календарі, а зайнятий час у Google Calendar враховується при підборі слотів.

Перерахунок слотів відбувається автоматично після будь-яких змін (вільний час, команди, бронювання) та
кожні 15 хвилин у фоновому режимі.

## Інтеграція з Google Calendar (опціонально)

Якщо інтеграція не потрібна — просто залиште `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` порожніми. Все інше
працюватиме без змін, лише без синхронізації з Google Calendar.

Щоб підключити:

1. Перейдіть у [Google Cloud Console](https://console.cloud.google.com/) → створіть (або оберіть) проєкт.
2. **APIs & Services → Library** → знайдіть і увімкніть **Google Calendar API**.
3. **APIs & Services → OAuth consent screen**:
   - Тип: External (або Internal, якщо є Google Workspace).
   - Додайте свій email як тестового користувача (поки додаток не верифіковано).
4. **APIs & Services → Credentials → Create Credentials → OAuth client ID**:
   - Тип застосунку: **Web application**.
   - **Authorized redirect URIs**: додайте `http://localhost:4000/api/calendar/oauth/callback`
     (має точно збігатися зі значенням `GOOGLE_REDIRECT_URI` у `.env`).
5. Скопіюйте **Client ID** та **Client secret** у `backend/.env`:
   ```
   GOOGLE_CLIENT_ID=...
   GOOGLE_CLIENT_SECRET=...
   GOOGLE_REDIRECT_URI=http://localhost:4000/api/calendar/oauth/callback
   ```
6. Перезапустіть backend. У кожного рекрутера в розділі «Профіль» з'явиться кнопка
   «Підключити Google Calendar» — після авторизації події співбесід будуть автоматично створюватись у
   його календарі.

При переході на хостинг потрібно буде:
- оновити `FRONTEND_URL` та `GOOGLE_REDIRECT_URI` на реальні домени;
- додати новий redirect URI у налаштуваннях OAuth-клієнта в Google Cloud Console;
- пройти верифікацію OAuth-застосунку в Google (якщо більше 100 користувачів / production-режим).

## Інтеграція з Google-таблицею відповідей форми (опціонально)

Кандидати спершу заповнюють Google-форму (відповіді складаються в Google-таблицю), а при записі вказують
лише email і Telegram. Якщо підключити таблицю, у «Загальному календарі» при кліку на співбесіду показуються
ПІБ, навчальна група та відповіді анкети — підтягнуті з таблиці за email/Telegram. Без підключення все працює,
просто без цих даних.

Доступ — через **сервісний акаунт** (сервер читає таблицю сам, не від імені рекрутера):

1. [Google Cloud Console](https://console.cloud.google.com/) → той самий (або новий) проєкт →
   **APIs & Services → Library** → увімкніть **Google Sheets API**.
2. **APIs & Services → Credentials → Create Credentials → Service account** → створіть акаунт
   (назва будь-яка, ролі не потрібні).
3. Відкрийте створений сервісний акаунт → вкладка **Keys → Add key → Create new key → JSON** →
   завантажиться JSON-файл із ключем. **Це і є секрет — не комітьте його.**
4. Скопіюйте email сервісного акаунта (виду `...@...iam.gserviceaccount.com`) і **поділіться з ним
   таблицею відповідей** (кнопка «Поділитися» у таблиці, право «Переглядач»).
5. Візьміть **ID таблиці** з її URL: `https://docs.google.com/spreadsheets/d/`**`<ОЦЕЙ_ID>`**`/edit`.
6. Заповніть `backend/.env`:
   ```
   GOOGLE_SHEETS_ID=<ID таблиці>
   # весь вміст JSON-ключа одним рядком (як є):
   GOOGLE_SERVICE_ACCOUNT_KEY={"type":"service_account","project_id":"...", ...}
   # АБО, замість рядка вище, шлях до файлу ключа:
   # GOOGLE_SERVICE_ACCOUNT_KEY_FILE=./service-account.json
   ```
   Колонки (email / Telegram / ПІБ / група) визначаються автоматично за заголовками. Якщо заголовки
   нестандартні — задайте їх явно: `GOOGLE_SHEETS_EMAIL_COLUMN`, `GOOGLE_SHEETS_TELEGRAM_COLUMN`,
   `GOOGLE_SHEETS_NAME_COLUMN`, `GOOGLE_SHEETS_GROUP_COLUMN` (див. `.env.example`). Решта колонок
   показуються як «питання → відповідь».

## Безпека

- Паролі зберігаються як bcrypt-хеші, авторизація — через JWT (термін дії `JWT_EXPIRES_IN`).
- Обов'язково задайте власний `JWT_SECRET` — сервер відмовиться запускатись зі значенням-заглушкою.
- Rate limiting: для входу та для публічного запису на співбесіду (захист від перебору та спаму бронювань),
  а також загальний ліміт на всі `/api` запити.
- Заголовки безпеки через `helmet`, CORS обмежено на `FRONTEND_URL`.
- Помилки сервера не повертають клієнту деталі/стек — лише загальне повідомлення (деталі логуються
  на сервері).
- Видалення рекрутера — «м'яке» (деактивація), дані бронювань та історія співбесід не втрачаються.
- **Перед виходом у продакшн**:
  - змініть усі паролі за замовчуванням (адмін + всі рекрутери);
  - використовуйте HTTPS;
  - зберігайте `.env` поза репозиторієм (вже додано у `.gitignore`-подібний підхід — не комітьте його);
  - за потреби перенесіть SQLite на Postgres для production-навантаження (структура запитів у
    `backend/src/db` ізольована в одному модулі, заміна БД не вимагає змін в інших файлах routes).

## Налаштування тривалості слотів

У таблиці `settings` (поле `slot_duration_minutes`) зберігається тривалість одного слота співбесіди в
хвилинах (за замовчуванням — значення `DEFAULT_SLOT_DURATION_MINUTES` з `.env`, тобто 45, що відповідає
сітці вибору вільного часу 08:00–21:00 з блоками «45 хв співбесіда + 15 хв перерва»). Змінити можна
безпосередньо в БД; за потреби можна додати UI для цього в адмінці.

## Запуск через Docker (найпростіше)

У корені проєкту є `docker-compose.yml`: nginx роздає зібраний фронтенд і проксує `/api` на backend,
SQLite зберігається у Docker-волюмі (`db-data`), тож переживає перезапуски.

```bash
cp .env.example .env        # заповніть JWT_SECRET та паролі
docker compose up -d --build
# відкрийте http://localhost:8080
```

Backend назовні не публікується — він доступний лише через проксі фронтенду, тож весь застосунок працює
за одним портом (`HTTP_PORT`, за замовчуванням 8080).

## Деплой — стабільно і безкоштовно

SQLite — це файл, тому хостинг має давати **постійний диск** і **не присипляти** процес. Нижче дві робочі
безкоштовні схеми.

### Варіант A (рекомендовано): безкоштовна VM + Docker + Cloudflare Tunnel

Найстабільніше та реально безкоштовне для невеликого проєкту.

1. Візьміть **завжди-безкоштовну VM**: Oracle Cloud *Always Free*, Google Cloud *e2-micro free tier* або
   будь-який інший «always free» інстанс. Встановіть на ній Docker.
2. Залийте код (`git clone`), створіть `.env`, запустіть `docker compose up -d --build`.
3. Для HTTPS і домену без відкривання портів — **Cloudflare Tunnel** (безкоштовний): встановіть `cloudflared`,
   `cloudflared tunnel ...` і вкажіть тунель на `http://localhost:8080`. Отримаєте `https://ваш-домен` з
   сертифікатом, а сама VM лишається закритою. (Альтернатива — Caddy на VM для авто-HTTPS.)
4. У `.env` задайте `FRONTEND_URL=https://ваш-домен`, а в Google Cloud — додайте
   `https://ваш-домен/api/calendar/oauth/callback` як redirect URI і встановіть це ж значення у
   `GOOGLE_REDIRECT_URI`.

### Варіант B (найпростіше кероване): Fly.io

Fly запускає Docker-образ і дає **persistent volume** для SQLite, не присипляє додаток і коштує близько нуля
на малому трафіку.

1. Встановіть `flyctl`, `fly launch` у теці `backend` (візьме `backend/Dockerfile`).
2. Створіть volume і змонтуйте його на `/data` (там лежить `data.sqlite` — змінна `DB_PATH`).
3. Задайте секрети: `fly secrets set JWT_SECRET=... ADMIN_PASSWORD=... FRONTEND_URL=...`.
4. Frontend зберіть окремо (`npm run build`) і роздайте через **Vercel / Netlify / Cloudflare Pages**
   (безкоштовно), вказавши проксування `/api` на адресу backend на Fly.

### Найпростіший «split» (з застереженням)

Frontend — на **Vercel/Netlify** (безкоштовно, стабільно). Backend — будь-де, але пам'ятайте:
на безкоштовних тарифах **Render/Railway** контейнер присипляється і **диск ефемерний** — SQLite буде
скидатись при перезапусках. Для такого шляху або під'єднайте постійний диск, або перенесіть БД на Postgres
(запити ізольовані в `backend/src/db`, заміна не потребує змін у роутах).

### Спільне для будь-якого хостингу
- `NODE_ENV=production`, унікальний `JWT_SECRET`, `FRONTEND_URL` = реальний домен фронтенду.
- HTTPS обов'язково (Cloudflare/Caddy/хостинг дають його безкоштовно).
- Оновіть `GOOGLE_REDIRECT_URI` на `https://<домен>/api/calendar/oauth/callback` і додайте його в OAuth-клієнт.
- Змініть усі паролі за замовчуванням після першого входу.
